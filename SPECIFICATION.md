# terraform-provider-coolify: Specification & Implementation Plan

## 1. Technology Stack (Researched)

All decisions below are based on studying bpg/proxmox (2k stars), tailscale/tailscale (best-in-class), SierraJC/coolify (existing community provider), and HashiCorp's official guidance.

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Language** | Go 1.24+ | Mandatory for Terraform providers |
| **Framework** | Terraform Plugin Framework (NOT SDK v2) | HashiCorp's official recommendation for all new providers. SDK v2 is in maintenance mode. |
| **API Client** | Hand-written Go HTTP client | SierraJC's Issue #29 shows they want to MOVE AWAY from OpenAPI codegen. Hand-written clients are more reliable with Coolify's imperfect OpenAPI spec. |
| **Testing** | `terraform-plugin-testing` + `net/http/httptest` mocks + acceptance tests | Three-tier: unit tests (mock HTTP), acceptance tests (real Coolify), and import tests. |
| **Documentation** | `tfplugindocs` (auto-generated from code) | Standard approach. Uses `MarkdownDescription` fields in schema. |
| **Linting** | `golangci-lint` with strict config | SierraJC skipped this (TODO). We enforce it from day one. |
| **CI/CD** | GitHub Actions (4 jobs: test, lint, format, docs-check) | Following Tailscale's proven pattern. |
| **Release** | GoReleaser + GPG signing | Standard for Terraform Registry publication. |
| **Project Structure** | `internal/` with service-based subpackages | Scalable pattern from AWS provider, not flat like Tailscale. |

## 4. Provider Configuration

```hcl
provider "coolify" {
  endpoint = "http://coolify.example.com:8000" # or COOLIFY_ENDPOINT
  token    = "your-api-token"                  # or COOLIFY_TOKEN (sensitive)
}
```

| Attribute | Type | Required | Env Var | Description |
|-----------|------|----------|---------|-------------|
| `endpoint` | string | Yes | `COOLIFY_ENDPOINT` | Coolify API base URL |
| `token` | string (sensitive) | Yes | `COOLIFY_TOKEN` | API bearer token |

## 5. Testing Strategy

### Unit Tests (every resource)
- Use `net/http/httptest` to create a mock Coolify API server
- Test all CRUD operations against mock responses
- Test error handling (404, 500, auth failures)
- Test schema validation
- Run with `make test` (no external dependencies)

### Acceptance Tests (every resource)
- Run against a real Coolify instance
- Gated behind `TF_ACC=1`
- Use `acctest.RandomWithPrefix("tf-acc")` for unique names
- Multi-step: create, update, import, destroy
- `CheckDestroy` verifies cleanup
- Run with `make testacc`

### Test-to-Feature Mapping
Every resource file has a matching `_test.go` with tests for:
1. Basic create and read-back
2. Update (modify attributes, verify plan)
3. Import (by ID)
4. Destroy and verify cleanup
5. Error cases (invalid input, not found)

## 6. Implementation Order

1. **Scaffold**: repo, go.mod, main.go, GNUmakefile, CI, .goreleaser.yml
2. **Client**: HTTP client with auth, retries, error handling
3. **Provider**: Configuration, schema, registration
4. **coolify_project**: Simplest resource, proves the pattern
5. **coolify_private_key**: Independent, needed by servers
6. **coolify_server**: Depends on private key
7. **coolify_application**: Most complex, most impactful
8. **coolify_environment_variable**: Critical for IaC workflows
9. **Data sources**: project(s), server(s), team
10. **Database resources**: postgresql, mysql, mariadb, redis, mongodb
11. **coolify_service**: One-click services
12. **coolify_deployment**: Trigger deployments
13. **Documentation**: Generate with tfplugindocs, add examples
14. **Release**: Tag v0.1.0, publish to Registry

## 7. Key Design Decisions

1. **Type-specific database resources** (not generic): `coolify_postgresql_database`, `coolify_mysql_database`, etc. More Terraform-idiomatic. Each DB type has different configuration options.

2. **Environment variables as a separate resource** (not nested in application): Allows managing env vars independently, supports bulk operations, cleaner state management.

3. **Hand-written API client** (not generated): Coolify's OpenAPI spec has known issues (Issue #6462, #7043). A hand-written client can handle quirks like Cuid2 IDs marked as UUID, GET-based start/stop endpoints, and incomplete response schemas.

4. **Retryable HTTP client**: Use `hashicorp/go-retryablehttp` for resilience against transient API failures.

5. **No mux (pure Framework)**: Unlike Tailscale's hybrid approach, we start fresh with Plugin Framework only. No SDK v2 dependency.
